## 2. Escopo funcional do Sistema de Monitoramento de Criptomoedas

Funcionalidades definidas (todas sustentadas nos requisitos obrigatórios):

1. **Autenticação** – registo, login, logout, recuperação de senha via email.
2. **Dashboard** – listagem das principais criptomoedas (preço, variação 24h, market cap) obtidas da API externa, com pesquisa por nome/símbolo.
3. **Detalhe de uma criptomoeda** – gráfico simples de variação, dados de mercado.
4. **Watchlist pessoal** – CRUD de favoritos (adicionar/remover moedas). Cada utilizador gere a sua lista.
5. **Alertas de preço** – criação, edição, remoção de alertas (preço alvo, direção). *(Esta funcionalidade garante uma terceira tabela com operações CRUD, além de users e watchlist.)*
6. **Exportação** – download da watchlist em CSV (ou PDF).
7. **Preferências** – alternância claro/escuro e seleção de idioma (PT/EN), guardadas localmente.

**Justificação para NÃO incluir tipos de utilizador:**  
O sistema é de uso pessoal. Não há operações administrativas que justifiquem uma separação de papéis. A omissão é intencional e coerente com o enunciado.

## 3. API externa escolhida

**CoinGecko API v3** (gratuita, sem chave obrigatória para os endpoints de mercado e detalhe).  
- Endpoints: `/coins/markets`, `/coins/{id}`, `/simple/price`  
- Limites generosos (~10–30 chamadas/minuto sem chave).  
- Dados fiáveis e bem estruturados.  
*(Não recorremos a web scraping – ver secção 11)*

---

## 4. Arquitetura geral

```
[Angular SPA] ---(HTTP/JSON)---> [PHP Backend (REST)] ---(HTTP)---> [CoinGecko API]
                                     |        |
                                  [MySQL]   [Email (SMTP)]
```

- O frontend **não comunica diretamente com a API externa**. O backend atua como proxy, centralizando caching e evitando exposição de chaves (se viessem a ser usadas).
- Autenticação via **JWT** (stateless).
- Backend organizado em camadas: **endpoints (controllers) → lógica de negócio (services) → acesso a dados (models/DB)**.

## 6. Base de dados (MySQL/MariaDB)

### 6.1 Tabelas e relacionamentos

```sql
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    language ENUM('pt','en') DEFAULT 'pt',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE watchlist (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    coin_id VARCHAR(50) NOT NULL,               -- ex.: 'bitcoin'
    added_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    UNIQUE KEY unique_user_coin (user_id, coin_id)
);

CREATE TABLE alerts (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    coin_id VARCHAR(50) NOT NULL,
    target_price DECIMAL(18, 8) NOT NULL,
    direction ENUM('above', 'below') NOT NULL,
    is_active TINYINT(1) DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE password_resets (
    email VARCHAR(100) NOT NULL,
    token VARCHAR(255) NOT NULL,
    expires_at DATETIME NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_email (email)
);
```

- **≥3 tabelas**: users, watchlist, alerts (4 com password_resets).  
- **CRUD**: users (registo, update), watchlist (add, list, delete), alerts (create, read, update, delete).  
- Integridade referencial com chaves estrangeiras.

## 7. Checklist de requisitos obrigatórios

| Requisito | Como será cumprido |
|-----------|-------------------|
| **Autenticação completa** | Endpoints PHP: `register`, `login`, `forgot-password`, `reset-password`. JWT + email via PHPMailer. |

- `/dashboard` — visão geral de mercado

## 9. Exportação de dados

- Endpoint `GET /api/export/watchlist?format=csv` retorna `Content-Type: text/csv`.  
- O `ExportController` busca a watchlist do utilizador autenticado e monta as linhas.  
- No frontend, um botão “Exportar CSV” faz `window.open(apiUrl)` para iniciar o download.

---

## 10. Internacionalização e temas

- **ngx-translate** carrega os ficheiros da pasta `assets/i18n/`.  
- No registo, o utilizador escolhe o idioma preferido (campo `language` na tabela `users`).  
- O tema é guardado em `localStorage`; o serviço `ThemeService` aplica a classe `dark-theme` ao `<body>`, e todas as cores são derivadas de variáveis CSS.

## 11. Web scraping e funcionalidades de IA

- **Web scraping**: não será usado. O enunciado pede integração com API externa e o CoinGecko oferece uma API pública robusta. Scraping viola termos de serviço, não é fiável e desrespeita o princípio de engenharia limpa.  
- **IA no sistema final**: não é exigida. O documento refere “desenvolvimento assistido por IA”, ou seja, apoio durante a codificação. **Adicionar uma funcionalidade de IA (ex.: previsão de preços) seria uma extensão opcional não requerida**, que não substitui nenhum dos requisitos obrigatórios. Caso o grupo deseje incluí‑la, pode‑o fazer como *extra*, mas a entrega base não depende disso.
