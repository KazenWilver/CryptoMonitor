Abaixo vai uma leitura fiel do **Lab #04** e, depois, uma proposta **completa e implementável** para o **Sistema de Monitoramento de Criptomoedas**, sem extrapolar o que o PDF não diz. Onde algo não está explicitamente definido, marco como **[bloqueado]** ou como **[pressuposto]**. 

## 1) O que o Lab 4 realmente exige

O laboratório não é só “fazer sites”; ele avalia sobretudo **qualidade estrutural**, **organização do código**, **separação de responsabilidades**, **reutilização**, **modularidade** e **uso crítico de IA** como apoio ao desenvolvimento, não como substituto do raciocínio técnico. 

Também existe uma exigência organizacional forte: o trabalho é em **grupos de até 4 elementos**, com **8 sistemas no total** por grupo: **4 do primeiro conjunto** e **4 do segundo conjunto**. Cada elemento deve ser responsável por **um sistema de cada conjunto**, e essa responsabilidade tem de ficar visível no **histórico de commits do GitHub**. A não entrega completa, sem justificação aceita, implica **nota zero** para o grupo. 

Para o teu caso, o tema escolhido — **Sistema de Monitoramento de Criptomoedas** — pertence ao **primeiro conjunto**. O PDF também diz que há liberdade na definição das funcionalidades específicas, desde que o sistema seja **utilizável**, **coerente** e com **fluxo lógico consistente**, resolvendo um problema real dentro do tema. 

## 2) Requisitos obrigatórios que o sistema tem de cumprir

Independentemente da funcionalidade escolhida, o sistema precisa obrigatoriamente de:

* **autenticação completa**: registo, login, logout e recuperação de senha;
* **base de dados relacional** com pelo menos **3 tabelas** e suporte a **CRUD**;
* integração com **pelo menos uma API externa**, quando aplicável;
* **interface responsiva**;
* **modo claro e modo escuro**, com alternância funcional;
* suporte a **múltiplos idiomas** com mínimo de **2 línguas**;
* funcionalidade de **exportação de dados ou relatórios** em **CSV ou PDF**. 

Quando a natureza do sistema justificar, também deve haver:

* distinção entre tipos de utilizadores;
* permissões diferenciadas de acesso e operação. 

E a stack não pode fugir ao que o PDF definiu: **Angular no frontend**, **PHP puro no backend**, **SQL relacional**, e comunicação via **HTTP com JSON**. No frontend, o uso de **componentes reutilizáveis**, **módulos** e separação entre **interface / lógica / serviços** deve ficar evidente. No backend, tem de existir separação entre **lógica de negócio**, **acesso a dados** e **endpoints**. 

## 3) Leitura técnica do tema “Monitoramento de Criptomoedas”

O PDF não impõe funcionalidades específicas para este tema; ele permite liberdade, desde que o sistema seja coerente e útil. 

Para este tema, a estrutura mais natural e consistente é esta:

* painel com preços e variação das criptomoedas;
* detalhe de cada moeda;
* lista de favoritos / watchlist;
* alertas de preço;
* filtros por moeda, mercado ou variação;
* exportação de relatórios;

Isto encaixa bem no objetivo do laboratório porque produz um sistema real, útil e com fluxo lógico claro: autentica-se o utilizador, escolhem-se moedas, observa-se o mercado, configuram-se alertas, e exportam-se relatórios. Essa escolha é uma **proposta técnica**, não uma exigência explícita do PDF.

### 4.3 Backend PHP puro

Separação recomendada:

* **Controllers**: recebem request e devolvem JSON;
* **Services**: regras de negócio;
* **Repositories**: acesso a dados SQL;
* **Models**: representação dos dados;
* **Middleware**: autenticação, autorização, CORS, validações transversais;
* **Validators**: validação de payload;
* **Helpers**: utilitários de resposta, datas, formatação.

Fluxo ideal:

`Angular -> HTTP JSON -> PHP Controller -> Service -> Repository -> SQL -> Response JSON`

Isto está alinhado com a exigência do lab de separação entre **endpoints**, **lógica de negócio** e **acesso a dados**.

## 5) Base de dados relacional proposta

O PDF exige pelo menos 3 tabelas; aqui faz sentido desenhar uma estrutura normalizada, sem redundância desnecessária. 

### 5.1 Tabelas principais

#### `users`

Guarda contas, autenticação e preferências.

Campos sugeridos:

* `id`
* `name`
* `email` (único)
* `password_hash`
* `role`
* `preferred_language`
* `theme`
* `created_at`
* `updated_at`

#### `cryptocurrencies`

Catálogo de moedas monitoradas.

Campos sugeridos:

* `id`
* `symbol`
* `name`
* `slug`
* `logo_url`
* `is_active`
* `created_at`
* `updated_at`

#### `crypto_prices`

Histórico temporal dos preços.

Campos sugeridos:

* `id`
* `cryptocurrency_id` (FK)
* `price_usd`
* `market_cap`
* `volume_24h`
* `change_24h`
* `captured_at`

### 5.2 Tabelas de apoio

#### `watchlists`

Moedas seguidas por cada utilizador.

Campos:

* `id`
* `user_id` (FK)
* `cryptocurrency_id` (FK)
* `created_at`

Constraint importante:

* `UNIQUE(user_id, cryptocurrency_id)`

#### `alerts`

Alertas configurados pelo utilizador.

Campos:

* `id`
* `user_id` (FK)
* `cryptocurrency_id` (FK)
* `target_price`
* `condition_type`
* `is_active`
* `triggered_at`
* `created_at`

#### `export_logs`

Registo de exportações.

Campos:

* `id`
* `user_id` (FK)
* `report_type`
* `format`
* `created_at`

### 5.3 Relacionamentos

* `users 1:N watchlists`
* `users 1:N alerts`
* `users 1:N export_logs`
* `cryptocurrencies 1:N crypto_prices`
* `cryptocurrencies 1:N watchlists`
* `cryptocurrencies 1:N alerts`

### 5.4 Normalização

A estrutura acima fica essencialmente em **3FN**:

* dados do utilizador não se repetem em watchlists/alerts;
* dados da moeda ficam numa tabela única;
* preços históricos ficam separados do catálogo de moedas;
* relações N:N são resolvidas por tabelas intermédias.

Isto reduz redundância e facilita manutenção, que é exatamente o tipo de qualidade estrutural que o lab valoriza.

## 7) APIs externas: o que faz sentido usar

O PDF só exige que exista **pelo menos uma API externa, quando aplicável**. Para um sistema de criptomoedas, isso faz sentido e é praticamente central. 

### Recomendação prática

Usar uma API externa para:

* preços em tempo real;
* capitalização de mercado;
* volume;
* variação 24h;
* histórico.

### Melhor abordagem técnica

Para este sistema, eu recomendaria **uma API pública e estável de dados de mercado** como fonte principal. Se quiseres enriquecer o sistema, podes acrescentar:

* uma segunda API para notícias;
* uma terceira para conversões cambiais, caso o sistema tenha suporte a moedas fiduciárias.

### Observação importante

Como o PDF não fixa uma API específica, a escolha exata da API é uma **decisão técnica do projeto**, não uma regra do laboratório.

## 8) Web scraping: permitido ou não?

**[bloqueado]** O PDF **não menciona scraping de forma explícita**, portanto não dá para afirmar que está autorizado como parte do projeto. O que o lab exige explicitamente é **integração com API externa**, quando aplicável. 

### Recomendação técnica

A melhor abordagem é:

* usar **API oficial ou pública** como fonte primária;
* evitar scraping como dependência central;
* usar scraping apenas como **fallback experimental**, se houver permissão explícita do site alvo e isso não violar os termos do serviço.

Para um projeto académico, a opção mais segura, estável e defendível na apresentação é **API**.

O PDF diz que o desenvolvimento será feito com **assistência de ferramentas de Inteligência Artificial**, mas deixa claro que isso é para apoio crítico ao desenvolvimento, não uma obrigação de colocar IA como funcionalidade do sistema. 

### Conclusão

* **IA no processo de desenvolvimento:** sim, é parte do lab.
* **IA como feature do sistema:** **não é obrigatória**.

A funcionalidade mais coerente seria algo como:

Para este lab, eu trataria IA no sistema como **opcional**. O foco principal deve continuar em:

# 5) Base de Dados Final

# 5.1 users

```sql
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(120) NOT NULL,
    email VARCHAR(150) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role ENUM('user', 'admin') DEFAULT 'user',
    preferred_language VARCHAR(10) DEFAULT 'pt',
    theme ENUM('light', 'dark') DEFAULT 'dark',
    reset_token VARCHAR(255),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

---

# 5.2 cryptocurrencies

```sql
CREATE TABLE cryptocurrencies (
    id INT PRIMARY KEY AUTO_INCREMENT,
    symbol VARCHAR(20) UNIQUE NOT NULL,
    name VARCHAR(100) NOT NULL,
    slug VARCHAR(100),
    logo_url TEXT,
    is_active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

---

# 5.3 crypto_prices

```sql
CREATE TABLE crypto_prices (
    id INT PRIMARY KEY AUTO_INCREMENT,
    cryptocurrency_id INT NOT NULL,
    price_usd DECIMAL(18,8),
    market_cap BIGINT,
    volume_24h BIGINT,
    change_24h DECIMAL(10,2),
    captured_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (cryptocurrency_id)
    REFERENCES cryptocurrencies(id)
    ON DELETE CASCADE
);
```

---

# 5.4 watchlists

```sql
CREATE TABLE watchlists (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    cryptocurrency_id INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id)
    REFERENCES users(id)
    ON DELETE CASCADE,

    FOREIGN KEY (cryptocurrency_id)
    REFERENCES cryptocurrencies(id)
    ON DELETE CASCADE,

    UNIQUE(user_id, cryptocurrency_id)
);
```

---

# 5.5 alerts

```sql
CREATE TABLE alerts (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    cryptocurrency_id INT NOT NULL,
    condition_type ENUM('above', 'below') NOT NULL,
    target_price DECIMAL(18,8) NOT NULL,
    is_active BOOLEAN DEFAULT TRUE,
    triggered_at TIMESTAMP NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id)
    REFERENCES users(id)
    ON DELETE CASCADE,

    FOREIGN KEY (cryptocurrency_id)
    REFERENCES cryptocurrencies(id)
    ON DELETE CASCADE
);
```

---

# 5.6 export_logs

```sql
CREATE TABLE export_logs (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,
    report_type VARCHAR(100),
    format ENUM('csv', 'pdf'),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (user_id)
    REFERENCES users(id)
    ON DELETE CASCADE
);
```

# 8) Melhor Estratégia de Autenticação

O PDF não obriga JWT, mas para Angular + PHP REST API a solução mais coerente é:

## JWT Authentication

Porque:

* funciona muito bem com SPA Angular;
* desacopla frontend/backend;
* facilita APIs REST;
* evita sessão PHP tradicional.

Fluxo:

```text
Login
→ backend gera JWT
→ frontend guarda token
→ interceptor adiciona Authorization header
→ middleware valida token
```

### Frontend:

# 21) Melhor Estratégia Geral para Tirar Boa Nota

O lab claramente valoriza:

* organização;
* arquitetura;
* separação de responsabilidades;
* clareza;
* evolução progressiva;
* reutilização;

* boa UX/UI;
* commits organizados;
* documentação consistente.
